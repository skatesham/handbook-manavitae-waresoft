    Guia de Uso do Manual de Arquitetura

    📖 Sobre o Manual

    O Angular Clean Architecture — Guia Polimórfico é um manual completo e escalável que documenta as melhores práticas implementadas no projeto ManaVitae.

    📊 Estatísticas

    • Linhas: 796
    • Palavras: ~2.872
    • Tamanho: Otimizado para Windsurf (< 12.000 tokens)
    • Versão: 2.0

    🎯 Objetivo

    Criar um template de guideline universal que:

    ✅ Atende projetos de qualquer tamanho (MVP → Enterprise)
    ✅ Mantém qualidade consistente em todas as fases
    ✅ É polimórfico: adapta-se sem reescrita
    ✅ É pragmático: baseado em código real, não teoria

    🚀 Como Usar

    Para Novos Desenvolvedores

    1. Leia Filosofia e Arquitetura em Camadas
    2. Consulte Guia de Decisão Rápida quando tiver dúvidas
    3. Use Exemplos Completos como referência
    4. Revise Checklist de Qualidade antes de PR

    Para Code Review

    1. Use Checklist de Qualidade como roteiro
    2. Verifique Métricas de Qualidade para limites
    3. Consulte seções específicas quando necessário

    Para Arquitetos

    1. Entenda Padrões de Escala para decisões
    2. Use Evolução da Arquitetura para planejamento
    3. Adapte conforme fase do projeto (MVP/Produto/Escala)

    🎨 Diferenciais

    1. Polimorfismo Arquitetural

    O manual não força uma única abordagem. Ele oferece 3 níveis de complexidade:

    • Pequeno: Simplicidade máxima, extrair apenas quando necessário
    • Médio: Separação clara de responsabilidades
    • Grande: Arquitetura completa com camadas bem definidas

    2. Baseado em Código Real

    Todos os exemplos são extraídos do projeto ManaVitae: - DailyProgressService: Serviço de domínio real - StorageService: Infraestrutura real - DateUtils: Utilitários reais - Componentes smart/dumb reais

    3. Decisões Rápidas

    Seção Guia de Decisão Rápida responde: - Onde colocar meu código? - Quando extrair? - Signal, Getter ou Computed?

    4. Métricas Objetivas

    Limites claros para: - Linhas por arquivo - Níveis de aninhamento - Complexidade ciclomática - Número de métodos/parâmetros

    📐 Princípios Aplicados

    Clean Architecture (Robert C. Martin)

    • Separação em camadas
    • Dependency Inversion
    • Independência de frameworks

    Domain-Driven Design (Eric Evans)

    • Serviços por domínio
    • Linguagem ubíqua
    • Bounded contexts

    SOLID Principles

    • Single Responsibility
    • Open/Closed
    • Liskov Substitution
    • Interface Segregation
    • Dependency Inversion

    Functional Programming

    • Imutabilidade (readonly, as const)
    • Funções puras (utils)
    • Composição sobre herança

    🎯 Casos de Uso

    Caso 1: Novo Feature

    Pergunta: "Onde criar o serviço de autenticação?"

    Resposta (Guia de Decisão Rápida):

    É infraestrutura técnica (HTTP, Storage, Auth)?
→ core/services/

    Caso 2: Componente Crescendo

    Pergunta: "Meu componente tem 80 linhas, devo quebrar?"

    Resposta (Métricas de Qualidade):

    Linhas/Componente: < 100 (Pequeno), < 200 (Médio)
→ Ainda OK para projeto pequeno/médio

    Caso 3: Signal vs Getter

    Pergunta: "Quando usar signal() vs get?"

    Resposta (Guia de Decisão Rápida):

    // Signal: Estado mutável
readonly count = signal(0);

// Getter: Derivado simples
get doubled() { return this.count() * 2; }
